## Summary

`Generacion-Codigo-Destino-MIPS` is a complete, end-to-end compiler for a small imperative language (`.chip`) designed for teaching *chip configuration / embedded systems* concepts. It is **Project #3** of the *Compilers & Interpreters* course (IC5701, TecnolГіgico de Costa Rica), and it reuses and extends the front-end built in Projects I and II.

The compiler is not a toy: it implements every classic phase of a real compiler вҖ” a JFlex-generated scanner, an LALR parser generated by Java CUP with **panic-mode error recovery**, a scope-aware **strongly-typed** semantic analyzer, an abstract syntax tree, a **three-address intermediate representation**, and a modular **MIPS code generator** with a register allocator, a heap-based object model, a stack calling convention, and a peephole optimization pass.

Roughly **5,200 lines of hand-written Java** across 74 source files (after removing auto-generated Javadoc boilerplate), plus a ~1,700-line CUP grammar and a ~260-line JFlex specification, all wired together by a Maven build that generates the scanner/parser, produces a runnable fat JAR, and runs a JUnit 5 suite in CI.

---

## Problem it solves

A team wants a lightweight imperative language for configuring chips and building embedded systems. This project delivers the **back-end (target-code generation)** of that language's toolchain. Concretely, given a `.chip` source file the compiler must:

1. Preserve and correct the lexical, syntactic and semantic scope of Projects I and II.
2. Decide whether the source can be *generated by the grammar*, considering grammar, syntax **and** semantics.
3. Report lexical, syntactic and semantic errors using **panic-mode recovery** (report as many errors as possible instead of aborting on the first).
4. Emit a MIPS assembly file that is **semantically equivalent** to the source and executable in QtSPIM вҖ” produced *from* the compiler's own intermediate code.

If the program is valid, the compiler writes the intermediate code (`.ic`) and the target code (`.asm`). If it is invalid, no code is generated and the errors are reported.

## Objectives

**General.** Build, in Java with JFlex and CUP, the MIPS target-code generator for the `.chip` language, starting from three-address code, so that the output is semantically equivalent to the source and runs in QtSPIM.

**Specific.**

- Preserve/correct the lexical, syntactic and semantic phases of the previous projects.
- Determine whether a source file is generated by the grammar (syntax + semantics).
- Report lexical/syntactic/semantic errors via panic-mode recovery.
- Generate three-address intermediate code as the basis of translation.
- Write a `.asm` file (same base name as the source) with equivalent MIPS.
- Add class support (structure and use) through all four phases and translate it to MIPS.
- Validate everything with an automated test suite.

## Features

| Area | What it does |
|------|--------------|
| **Lexing** | JFlex scanner with line/column tracking, comment state handling (including *unterminated multiline comment* detection), and per-token reporting. |
| **Parsing** | LALR grammar (Java CUP). 38 benign shift/reduce conflicts resolved by shift, managed via `-expect 38`. |
| **Error recovery** | True **panic mode**: dozens of `error` productions synchronize to safe points (`:|`, `!`), plus a heuristic that *inserts* a missing statement terminator (`!`) to keep parsing. |
| **Semantics** | Scope-stacked symbol table, strong typing, boolean-only conditions, mandatory-`return` checking, single-`__main__` enforcement. |
| **Types** | `int`, `float`, `bool`, `char`, `string`. |
| **Expressions** | Arithmetic, relational and logical operators with correct precedence (incl. `^` power, `%` modulo). |
| **Control flow** | `if/else`, `do-while`, `switch/case/default`, `break`. |
| **Functions** | Parameters, return values, mandatory `__main__` procedure. |
| **Arrays** | One-dimensional arrays (declaration, indexed store/load). *(2-D arrays are parsed but not fully supported вҖ” see [Limitations](#limitations).)* |
| **I/O** | `cin` / `cout` mapped to MIPS syscalls. |
| **Classes** | Field layout, `new` with heap allocation (`sbrk`, syscall 9), field access by offset, `this`, constructors, and **static method dispatch**. |
| **IR** | Readable three-address code, written to `.ic`. Constants are intentionally **not** folded, keeping every temporary and operation visible. |
| **MIPS backend** | `.data`/`.text` emission, integer and floating-point paths, stack calling convention, and a **peephole optimizer** (redundant `sw`/`lw` вҶ’ `move`). |
| **Tooling** | Maven build (auto-generates scanner/parser), runnable fat JAR (shade), JUnit 5 suite, GitHub Actions CI. |

## Technical concepts applied

- **Compiler theory:** finite-automata lexing, **LALR parsing**, panic-mode error recovery, AST construction, **three-address intermediate representation**, target-code generation.
- **Semantic analysis:** scope-stacked symbol table (`Stack`-like of hash maps), strong static typing, type inference over expression trees, control-flow/return-path checks.

- **Design patterns:** Pipeline/phases, FaГ§ade (`Compilador`, `GeneradorMIPS`), Composite (AST), Visitor-style traversal, Strategy-style per-operation translators, DTO/Result objects, Builder-style emitters.
- **Software engineering:** SRP-driven package decomposition, generics & collections, functional callbacks (`Consumer` for error sinks, method references for label factories), build automation, **CI/CD**, parameterized testing.

cd Generacion-Codigo-Destino-MIPS/programa

The build runs JFlex and CUP to generate the scanner/parser under `target/generated-sources/`, compiles everything, runs the tests, and packages a runnable JAR with `maven-shade-plugin`.

## Execution flow

1. **CLI (`Main`)** validates arguments and the source path, creates the output directory.
2. **`Compilador.compilar`** opens the source as UTF-8, builds the `MiLexer`, and runs the CUP `Parser` (semantic analysis is interleaved during parsing).
3. If parsing fails fatally, the remaining token stream is drained so token/error reports stay complete.
4. **Acceptance** is the conjunction of: syntax complete **AND** no lexical errors **AND** no syntactic errors **AND** no semantic errors.
5. Only if accepted: **IR generation** walks the AST into three-address code, then the **MIPS generator** analyzes the IR (`AnalizadorIRMIPS`), emits `.data`/`.text`, translates each instruction through the specialized translators, and runs the peephole pass.
6. **Report writers** persist tokens, symbol table, errors, verdict, `.ic` and `.asm`.

## Design decisions

- **Two-tier IR before MIPS.** Generating three-address code first (rather than emitting MIPS directly from the AST) isolates machine-specific concerns in the backend and makes the pipeline testable at the IR level. *(Required by the assignment and leveraged deliberately.)*
- **No constant folding in the IR.** Kept intentionally verbose so temporaries and operator precedence are visible in `.ic` вҖ” better for grading/inspection and for validating the backend.
- **Modular backend.** `GeneradorMIPS` is a faГ§ade dispatching on the IR operation to focused translators (`TraductorMemoria`, `TraductorOperaciones`, `TraductorControl`, `TraductorIO`, `TraductorFunciones`, `TraductorObjetos`, `TraductorTransferencia`). Each is independently readable and swappable.
- **Panic-mode over fail-fast.** The parser is engineered to keep going and report *many* errors, using both grammar `error` productions and a terminator-insertion heuristic.
- **Object model with a reserved vtable slot.** Word 0 of every object is reserved for a future vtable; fields start at offset 4 вҖ” a forward-looking layout even though dispatch is currently static.
- **Deterministic build with pinned tool versions** and `-expect 38` to catch accidental grammar changes.

## Challenges encountered

- **Grammar conflicts.** Reaching a workable LALR grammar with a distinctive syntax (`|: :|`, `<| |>`, `<< >>`, `~`, `!`) produced 38 shift/reduce conflicts that had to be understood and accepted as benign.
- **Error recovery.** Designing synchronization points and the `!`-insertion heuristic so the parser recovers without cascading spurious errors.
- **MIPS object model.** Mapping OO semantics (heap allocation, `this`, offsets, constructors, static dispatch) onto MIPS with a stack calling convention.
- **Float vs int paths.** Maintaining two register/instruction families (`$t*`/`add` vs `$f*`/`add.s`) consistently through the backend.

---

## Limitations

- **2-D arrays** from the assignment's own example are currently **rejected** (only 1-D arrays translate end-to-end).
- **Classes:** no inheritance or polymorphism; field access inside methods is explicit `this.field`; fields must be declared before methods; no object-typed fields/params/returns (except `this`); no `free`; no null-pointer checks.
- **Register allocator** is a 6-register free-list (`$t0вҖ“$t5`) with **no spilling** вҖ” very complex expressions raise an internal error instead of spilling to memory.
- **Temporaries are global `.data` labels**, so the current model is **not recursion-safe** (a recursive call would clobber a caller's temporaries). *This limitation is documented in-code in `AnalizadorIRMIPS.construirTablaDirecciones` and `TraductorFuncionesMIPS`.*

---

## Possible improvements

- Spill registers to the stack (or add a linear-scan allocator) instead of failing when `$t0вҖ“$t5` are exhausted.
- Move temporaries/locals into per-call **stack frames** to enable recursion and reentrancy.
- Finish **2-D arrays** end-to-end and add bounds handling.
- Replace the string-based peephole optimizer with a pass over structured instructions; add more peepholes (dead-store elimination, constant propagation).
- Unit-test individual translators, the optimizer and the register allocator, not just whole-program integration.

## Future work

Inheritance & dynamic dispatch (the vtable slot is already reserved) В· object composition В· a small standard library В· optional constant-folding/optimization levels В· a golden-file testing harness for `.asm` output.

## Lessons learned

- A clean phase boundary (AST вҶ’ IR вҶ’ MIPS) pays off: the backend became testable and the object model was added without disturbing the front-end.
- Error recovery is a *design* problem, not an afterthought вҖ” synchronization points must be chosen deliberately.
- Small, single-responsibility translators are far easier to reason about than a monolithic generator.
- Choosing where **not** to optimize (no IR constant folding) can be as valuable as optimizing, when inspectability is the goal.
